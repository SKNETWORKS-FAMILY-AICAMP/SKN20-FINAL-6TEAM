# LLM 도메인 분류 실패 분석 및 개선 제안

> **분석 대상**: State E (LLM ON) 평가 결과 (`answers_state_e_llm_on.json`)
> **비교 기준**: State E (LLM OFF) — 동일 코드에서 키워드 분류 사용
> **분석일**: 2026-03-10

---

## 1. 요약

LLM 도메인 분류 활성화 시 **유효 질문 70건 중 15건(21.4%)이 Answer Relevancy = 0.0**으로 사실상 실패했다.
이는 State E(LLM OFF) 대비 전 메트릭 하락의 주된 원인이며, 크게 3가지 패턴으로 분류된다.

| 실패 유형 | 건수 | 비율 | 핵심 원인 |
|-----------|------|------|----------|
| **오거부 (False Rejection)** | 8건 | 11.4% | LLM이 유효 질문을 `is_relevant=false`로 판정 |
| **검색 성공 + 답변 실패** | 7건 | 10.0% | 컨텍스트는 좋았으나 답변 생성 품질 저하 |
| **검색 자체 실패** | 3건 | 4.3% | 벡터DB에 관련 문서 부재 또는 도메인 오분류 |

## 2. 현재 라우터 도메인 분류 흐름

```
사용자 질문
  │
  ▼
LLM Query Rewrite (대화 이력 있으면)
  │
  ▼
DomainClassifier.classify(rewritten_query, original_query)
  │  ├─ 1단계: 치챗 감지 (original_query 기준)
  │  └─ 2단계: 도메인 분류 (rewritten_query 기준)
  │
  ▼
┌─────────────────────────────────────────┐
│ is_relevant = false                     │
│  ├─ intent=chitchat_* → 치챗 응답       │
│  ├─ method=llm_retry_failed → 오류 응답 │
│  └─ 그 외 → 거부 응답 (검색 안 함)      │
├─────────────────────────────────────────┤
│ is_relevant = true                      │
│  └─ domains=[...] → 도메인별 에이전트   │
│     → 벡터 검색 → 답변 생성             │
└─────────────────────────────────────────┘
```

**핵심**: `is_relevant=false`이면 검색 자체를 건너뛰고 즉시 거부 응답을 반환한다.

## 5. 근본 원인

### 5-1. LLM 분류 프롬프트의 도메인 키워드 커버리지 부족

현재 `LLM_DOMAIN_CLASSIFICATION_PROMPT`의 도메인 예시:

```
- startup_funding: 창업, 사업자등록, 법인설립, 지원사업, 보조금, 투자유치, 마케팅, 프랜차이즈
- finance_tax: 세금, 부가세, 법인세, 회계, 세무, 결산, 연말정산, 원천징수
- hr_labor: 근로, 채용, 급여, 퇴직금, 연차, 4대보험, 근로계약, 취업규칙, 프리랜서, 용역
- law_common: 법률, 소송, 특허, 상표, 저작권, 상법, 민법, 계약법, NDA, 비밀유지계약
```

**누락된 키워드** (오거부된 질문에서 추출):

| 도메인 | 누락 키워드 |
|--------|-----------|
| startup_funding | 정책자금, 벤처인증, 벤처기업, 사회적기업, 소상공인 |
| finance_tax | 종합소득세, 부가가치세(풀네임), 간이과세, 일반과세 |
| hr_labor | 출산휴가, 배우자출산휴가, 육아휴직, 외국인근로자, 고용허가, 비자 |

### 5-2. `is_relevant=false` 판정이 과도하게 공격적

LLM이 확신이 없을 때 `is_relevant=false`로 거부하는 경향 → 안전하지만 recall 손실.
키워드 분류는 매칭되는 키워드가 하나라도 있으면 해당 도메인으로 라우팅하여 recall 유지.

### 5-3. 키워드 분류와 LLM 분류의 fallback 부재

현재 **순수 LLM 기반** 시스템으로, LLM이 실패하면 바로 거부.
키워드 기반 fallback이 없어 LLM 판단 오류가 그대로 최종 결과에 반영됨.

## 6. 개선 제안

### 제안 1: 하이브리드 분류 (LLM + 키워드 Fallback) ⭐ 최우선

```
사용자 질문
  │
  ▼
LLM 도메인 분류
  │
  ├─ is_relevant=true, confidence ≥ 0.7 → LLM 결과 사용
  │
  ├─ is_relevant=false 또는 confidence < 0.7
  │   └─ 키워드 분류 Fallback 실행
  │       ├─ 키워드 매칭 성공 → 키워드 결과 사용 (검색 진행)
  │       └─ 키워드 매칭 실패 → 거부 응답
  │
  └─ method=llm_error → 키워드 분류 Fallback
```

**기대 효과**: 오거부 8건 중 대부분 복구 가능. LLM OFF에서 정상 처리되던 질문이므로 키워드 fallback으로 충분.

### 제안 2: LLM 분류 프롬프트 키워드 보강

`LLM_DOMAIN_CLASSIFICATION_PROMPT`에 아래 키워드 추가:

```
- startup_funding 추가: 정책자금, 벤처기업, 벤처인증, 사회적기업, 소상공인,
                        수출바우처, 소공인, 예비창업자, 1인창업
- finance_tax 추가: 종합소득세, 부가가치세, 간이과세, 일반과세, 근로장려금,
                    세액감면, 세액공제, 전자세금계산서
- hr_labor 추가: 출산휴가, 배우자출산휴가, 육아휴직, 외국인근로자, 고용허가,
                비자(취업), 산업재해, 산재, 수습, 인턴
- law_common 추가: 개인정보, 개인정보보호법, 하도급, 상가임대차, 청약철회,
                   채권회수, 지급명령
```

**기대 효과**: 오거부율 감소. 단, 프롬프트만으로는 LLM의 일관성 문제를 완전히 해결하기 어려움.

### 제안 3: 거부 임계값(Confidence Threshold) 완화

현재 LLM이 `is_relevant=false`를 반환하면 무조건 거부. 대신:

```python
# 현재
if not classification.is_relevant:
    return REJECTION_RESPONSE

# 개선안
if not classification.is_relevant and classification.confidence >= 0.8:
    return REJECTION_RESPONSE  # 높은 확신의 거부만 수용
elif not classification.is_relevant and classification.confidence < 0.8:
    # 확신이 낮으면 키워드 fallback 또는 검색 진행
    fallback_domains = keyword_classify(query)
    if fallback_domains:
        classification.domains = fallback_domains
        classification.is_relevant = True
```

**기대 효과**: LLM이 애매하게 거부한 케이스(confidence 0.3~0.7)를 복구.

### 제안 4: 도메인별 하이브리드 전략

법률 도메인은 LLM 분류가 키워드보다 우수(Context Recall +0.12), 나머지는 키워드가 우수.

```python
DOMAIN_CLASSIFICATION_STRATEGY = {
    "law_common": "llm_preferred",      # LLM 우선, 키워드 보조
    "startup_funding": "keyword_preferred",  # 키워드 우선, LLM 보조
    "finance_tax": "keyword_preferred",
    "hr_labor": "keyword_preferred",
}
```

**기대 효과**: 도메인별 강점을 살린 최적 분류.

### 제안 5: 멀티 도메인 예산 분배 개선

복합 질문(Q49, Q66, Q67, Q68)에서 실패율이 높음. 현재 `primary_domain_budget_ratio`로 주 도메인에 예산 집중 → 보조 도메인 문서 부족.

- 2개 도메인: 6:4 → **5:5** (균등 분배)
- 3개 도메인: 주 도메인 50% → **각 도메인 최소 retrieval_k=2 보장**

## 7. 우선순위 및 예상 효과

| 순위 | 제안 | 구현 난이도 | 예상 효과 |
|------|------|-----------|----------|
| 1 | 하이브리드 분류 (LLM + 키워드 Fallback) | 중 | 오거부 8건 → 0~2건 |
| 2 | 프롬프트 키워드 보강 | 하 | 오거부 3~5건 감소 |
| 3 | 거부 임계값 완화 | 하 | 애매한 거부 2~3건 복구 |
| 4 | 도메인별 하이브리드 전략 | 중 | 전체 메트릭 최적화 |
| 5 | 멀티 도메인 예산 분배 | 하 | 복합 질문 3~4건 개선 |

**제안 1+2 조합 적용 시**, LLM ON 상태에서도 LLM OFF 수준 이상의 성능 달성 가능.
